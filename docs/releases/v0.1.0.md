# openLight v0.1.0

`v0.1.0` is the release where `openLight` becomes a Telegram-first monitoring loop: define watches once, receive actionable alerts, and handle safe recovery for allowlisted services and containers from chat.

## Good Fit For

- Raspberry Pi and homelab users who want persistent service and host alerts in Telegram
- Operators who want safe restart, logs, and status actions without standing up a full monitoring stack
- Self-hosters who want local Ollama by default, but still want deterministic-only mode or an easy OpenAI switch

## Highlights

- Added persistent watches for services, CPU, memory, disk, and temperature
- Added alert actions and Telegram inline buttons: `Restart`, `Logs`, `Status`, and `Ignore`
- Added `/enable docker`, `/enable system`, and `/enable auto-heal` packs
- Added a one-command installer that resolves the latest tagged release by default
- Improved LLM fallback with lower-latency routing, heuristic rescue, and profile switching via `LLM_PROFILE`
- Improved smoke flows and Raspberry Pi deployment helpers for Ollama and OpenAI profile checks
- Reworked the docs around the monitoring workflow and added a Telegram demo GIF

## Try It

Use `v0.1.0` if you want the first tagged release where `openLight` can both watch and react.

```bash
export TELEGRAM_BOT_TOKEN=123456:replace-me
export ALLOWED_USER_IDS=111111111
export OPENLIGHT_REF=v0.1.0
curl -fsSL https://raw.githubusercontent.com/evgenii-engineer/openLight/v0.1.0/scripts/install.sh | bash
```

After it starts, try:

```text
/start
/enable system
/watch add service tailscale ask for 30s cooldown 10m
```

## Known Limitations

- Metric watches currently support `notify` only; `ask` and `auto` are for service-down watches
- Running inside Docker still does not automatically grant host-level file or service access
- The primary interface is Telegram; the CLI remains operator- and test-oriented
- The project is still `v0.x` and intentionally narrow in scope
