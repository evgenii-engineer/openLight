# openLight v0.3.0

`v0.3.0` introduces an edge/brain network topology that lets Raspberry Pi
edge nodes offload LLM inference, voice transcription, and skill execution
to a central brain node over a lightweight HTTP API. Voice handling is
tightened, the `/status` command is wired through a new runtime hooks
layer, and a `think` skill exposes an explicit reasoning step through
the normal routing pipeline.

Nothing about the project's posture changes. openLight is still a small,
deterministic, local-first infrastructure tool — the LLM is still optional,
still allowlist-bound, and never expands the surface on its own. The
edge/brain topology is entirely opt-in.

## Good fit for

- Raspberry Pi operators who want to offload LLM calls to a Mac mini or
  server without running Ollama on the Pi itself
- Homelab operators who want a display dashboard on a Pi with a screen
- Anyone on `v0.2.0` who needs better voice handling or a reliable
  `/status` command

## Highlights

- **Edge/brain network topology.** A `brain` node (Mac mini or server)
  exposes an HTTP API; `edge` nodes (Raspberry Pi) route LLM calls, voice
  transcription, and skill invocations to it via new `remote` providers.
  New example configs (`configs/agent.brain.example.yaml`,
  `configs/agent.edge.example.yaml`) document the full setup.
- **Remote LLM provider.** `internal/llm/remote.go` — edge nodes send
  classification and chat requests to the brain over HTTP instead of calling
  Ollama locally.
- **Remote voice transcriber.** `internal/voice/remote.go` — voice notes
  received on an edge node are transcribed on the brain.
- **Remote skill proxy.** `internal/skills/remote.go` — skills that require
  local brain resources (heavy models, display hardware) can be proxied from
  an edge node transparently.
- **Display skill and Pi dashboard.** `internal/skills/display/skill.go`
  plus a 618-line Python dashboard (`scripts/display-dashboard.py`) and a
  deploy script (`scripts/deploy-rpi-display.sh`) for Pi nodes with a
  screen. Ships with a systemd unit
  (`deployments/systemd/openlight-display.service`).
- **Think skill.** `internal/skills/chat/think_skill.go` exposes an explicit
  reasoning step through the routing pipeline.
- **Voice improvements.** Config validation extended for `whisper_path`,
  `model_path`, and `language`. Telegram voice-message client tightened.
  Russian-language routing tests added. `openlight doctor` gains
  voice-specific probes.
- **Status command via hooks.** `/status` is now wired through a new
  runtime hooks layer (`internal/runtime/hooks.go`) instead of a direct
  call path, making it easier to extend without touching the runtime core.

## Try it

```bash
export TELEGRAM_BOT_TOKEN=123456:replace-me
export ALLOWED_USER_IDS=111111111
export OPENLIGHT_REF=v0.3.0
curl -fsSL https://raw.githubusercontent.com/evgenii-engineer/openLight/v0.3.0/scripts/install.sh | bash
```

Once it starts, try:

```text
/start
/status
think what services should I watch on a Pi homelab?
```

For the edge/brain setup, copy the example configs, fill in the shared
token, and start the brain node first:

```bash
cp configs/agent.brain.example.yaml agent.brain.yaml
# edit agent.brain.yaml — set brain.token
openlight agent --config agent.brain.yaml
```

Then point each Pi at the brain:

```bash
cp configs/agent.edge.example.yaml agent.edge.yaml
# edit agent.edge.yaml — set brain.endpoint and brain.token
openlight agent --config agent.edge.yaml
```

## Migration notes from v0.2.0

Existing `v0.2.0` configs work unchanged. All new surfaces are opt-in.

- **Edge/brain topology is opt-in.** Nothing in a standard single-node
  config needs to change. To use the brain as an LLM backend, add
  `llm.provider: remote` and `brain.endpoint` / `brain.token` to your
  edge config.
- **Remote voice is opt-in.** Add `voice.provider: remote` to delegate
  transcription to the brain. The local `whisper-cli` path continues to
  work unchanged.
- **Think skill is registered automatically.** No config change required;
  it appears in `/skills` alongside other chat skills.
- **Display dashboard requires Python 3 on the Pi.** Run
  `scripts/deploy-rpi-display.sh` to install dependencies. The script is
  idempotent.
- **Status command behaviour is unchanged.** The hooks refactor is
  internal; the Telegram output is the same.

## Known limitations

- Metric watches still support `notify` only. Service-down watches are
  the ones that take `ask` / `auto` and offer action buttons.
- Running inside Docker still does not grant host-level file or service
  access automatically. Mount what you need, or reach hosts via `nodes:`.
- The brain HTTP server has no built-in TLS termination; put it behind
  a reverse proxy or restrict access to your local network.
- The display dashboard requires a framebuffer or display attached to the
  Pi; it does not run headless.
- The project is still `v0.x` and intentionally narrow in scope.
